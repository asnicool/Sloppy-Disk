# Plan: Preserve Album Selection & Search Results on Navigation

## Problem Analysis

When navigating away from the albums list or search results and returning via "back":
- **AlbumsView**: With random sort, albums are reshuffled each time (backend fetches new random selection)
- **SearchView**: Search results are lost, requiring a new search

Root cause: The current implementation fetches new data on every view mount, without any state preservation.

---

## Solution 1: Query Parameters (URL-driven)

**Approach**: Store filter state in URL query parameters.

- `/albums?sort=random&page=2` - preserves sort mode and pagination
- `/search?q=rock&page=1` - preserves search query
- On mount: Read URL params and fetch with those parameters

**Pros**:
- Browser back/forward works correctly
- Shareable URLs
- No extra state management needed
- Works with current backend API

**Cons**:
- Adds API calls on every navigation (no cache benefit)
- Random still reshuffles (backend creates new random selection)

---

## Solution 2: Client-side Cached State (Recommended)

**Approach**: Keep fetched data in Pinia store with a "stale" flag.

**Implementation**:
1. **AlbumsView**: 
   - Store albums in `mpdStore.albumsCache` keyed by `{sort, page}`
   - On mount: Check if cached data exists and is not stale (e.g., < 5 min old)
   - If fresh, use cache; otherwise fetch fresh

2. **SearchView**:
   - Store `mpdStore.lastSearchQuery` and `mpdStore.searchResultsCache`
   - On mount: Check if search query matches previous, reuse results

3. **Stale detection**:
   - Add timestamp to cached data
   - Optionally: Add a "stale" flag when navigating away that gets cleared on new fetch

**Pros**:
- True preservation - same albums shown on back navigation
- Better UX - instant load from cache
- Random sort works correctly
- Search results preserved

**Cons**:
- Slightly more complex state management
- Need to handle cache invalidation

---

## Recommendation: Solution 2

**Why it's smarter**:
1. Actually solves the problem - random albums won't reshuffle
2. Instant page load from cache (better UX)
3. Search results truly preserved
4. Browser back/forward works naturally
5. Minimal changes to existing architecture

**Files to modify**:
- `frontend/src/stores/mpdStore.js` - Add cache objects with timestamps
- `frontend/src/views/AlbumsView.vue` - Check cache before fetching
- `frontend/src/views/SearchView.vue` - Check cache before fetching
