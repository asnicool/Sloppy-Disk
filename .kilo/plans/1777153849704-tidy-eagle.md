# Drag-and-Drop Queue Management - Code Review & Bug Analysis

## High-Level Architecture

The queue page displays the playlist as a horizontal strip of album cards. Each card groups consecutive tracks sharing the same `album+artist` key. Two levels of drag-and-drop are supported:

- **Album-level**: drag album cards via the `.album-handle` grip, handled by `vuedraggable` in `QueueAlbumStrip.vue`
- **Track-level**: drag individual tracks within/between album cards, handled by `vuedraggable` in `QueueTrackList.vue`

Moves are sent to MPD via `POST /api/playlist/move`, then the entire playlist is re-fetched from MPD to update the UI.

---

## Drag-and-Drop Cases

### Case 1: Album Move (album card drag)
**Code**: `QueueAlbumStrip.vue:301-318` (`handleAlbumChange`)

When an album card is dragged from visual index `oldIndex` to `newIndex`:
1. Reads the original `groups` array from the `groupedPlaylist` computed (derives from `mpdStore.playlist`)
2. Manually reconstructs the post-move ordering: `tempGroups` = original groups with the moved group spliced to `newIndex`
3. Calculates `targetPos` = sum of `tracks.length` of all groups before `newIndex` in `tempGroups`
4. Calls `mpdStore.moveAlbum(start, movedGroup.tracks.length, targetPos)` → `MoveRange(start, start+length, targetPos)` on MPD

**MPD semantics**: `move START:END TO` removes the range, then inserts at position TO in the shortened playlist. TO past the end appends.

### Case 2: Track Move Within Same Album
**Code**: `QueueTrackList.vue:107-119` (`handleChange` with `event.moved`)

- `from = element.pos` (global position)
- `to = groupStartPos + newIndex` (destination global position)
- Calls `mpdStore.moveTrack(from, to)` → `Move(from, to)` on MPD

### Case 3: Track Move Across Albums
**Code**: `QueueTrackList.vue:114-118` (`handleChange` with `event.added`)

Same calculation as Case 2, but the track arrives in a different album's `QueueTrackList`. The `groupStartPos` is the destination album's start position.

### Case 4: Drop on Trashcan (album)
**Code**: `QueueAlbumStrip.vue:407-419`

Calls `removeAlbum(group)` which iterates from highest to lowest position, calling `mpdStore.removeFromPlaylist(pos)` for each track.

### Case 5: Drop on Trashcan (single track)
**Code**: `QueueAlbumStrip.vue:413-414`

Calls `mpdStore.removeFromPlaylist(draggedTrackPos)`.

### Case 6: "Clear Played" Button
**Code**: `QueueAlbumStrip.vue:335-358`

Collects all positions from fully-played albums, then calls `mpdStore.removeMultipleFromPlaylist(positions)` (sequential descending removals, single fetch at end).

## Bug Analysis

### BUG 1 (Critical): `removeAlbum` race condition — parallel un-awaited deletes

**Location**: `QueueAlbumStrip.vue:329-333`

```js
const removeAlbum = (group) => {
  for (let i = group.tracks.length - 1; i >= 0; i--) {
    mpdStore.removeFromPlaylist(group.startPos + i)  // async, NOT awaited
  }
}
```

Each `removeFromPlaylist` is async (sends HTTP DELETE then `fetchPlaylist()`). Since the loop doesn't `await`, **all delete requests fire simultaneously**. If they reach the backend out of order (e.g., pos 5 processed before pos 7), positions shift mid-operation and **wrong tracks get deleted**.

**Example**:
- Album at positions 5, 6, 7
- Requests fire: DELETE /5, DELETE /6, DELETE /7 (all at once)
- If DELETE /5 arrives first: position 6→5, 7→6
- DELETE /6 now removes what was originally at position 7
- DELETE /7 may fail or delete the wrong track entirely

**Impact**: Triggered by both the "X" button on album cards (line 101) and the trashcan drop zone (line 411). Can corrupt the playlist and delete arbitrary tracks.

**Fix**: Use `removeMultipleFromPlaylist` (which correctly awaits each descending deletion), or add `await` to the loop. The `removePlayedAlbums` function already does this correctly.

---

### BUG 2 (Moderate): Cross-drag race condition — mutated cached data used for album move targetPos

**Location**: `QueueAlbumStrip.vue:301-318` (`handleAlbumChange`)

`QueueTrackList` uses `:list="tracks"` which gives vuedraggable a direct reference to the `group.tracks` array. When a track is dragged between albums, vuedraggable **mutates these arrays in place** (splices the source, pushes to destination).

If the user drags a track and then **immediately** drags an album (before the track move's `fetchPlaylist()` completes):
- `mpdStore.playlist` is still the old data → `groupedPlaylist` computed returns the cached result
- But the cached result's `tracks` arrays have been mutated by the prior track drag
- `handleAlbumChange` reads `tempGroups[i].tracks.length` from these mutated arrays
- The `targetPos` calculation is off by the number of cross-dragged tracks

**Impact**: Requires two drags in very rapid succession. The resulting album move targets the wrong position.

---

### BUG 3 (Low/Moderate): Visual flickering — empty computed setter + no optimistic updates

**Location**: `QueueAlbumStrip.vue:293-295`

```js
set(newGroups) {
  // empty!
}
```

When vuedraggable reorders album cards, it calls the setter (no-op). Then:
1. vuedraggable shows the album at its new position (its internal DOM state)
2. The computed getter still returns the original order (since `mpdStore.playlist` hasn't changed)
3. Vue re-renders, reverting the visual to the old order
4. `moveAlbum` completes, `fetchPlaylist()` updates the playlist, computed re-evaluates
5. The album jumps to the correct new position

The user sees a brief "snap back then jump" flicker on every album drag.

---

### BUG 4 (UX): Cross-album track drag breaks album grouping

**Location**: `QueueTrackList.vue:114-118`

When a track from Album A is dropped into Album B's track list:
- The MPD `move` command is correct (moves the track to the right global position)
- But album grouping is **metadata-based** (album+artist key), not positional
- The moved track retains its original album metadata → it creates a new single-track "album" group or splits the destination album
- Visually: the track appears inside Album B's card momentarily (vuedraggable), then on playlist refresh it pops out into its own group

This is a design-level issue. Cross-album track drags produce a valid playlist state but a confusing visual result.

---

### BUG 5 (Edge Case): Stale `draggedItem` on trashcan drop

**Location**: `QueueAlbumStrip.vue:361-368` (capture) and `407-419` (use)

`draggedItem` is captured at drag start with the group's `startPos`. If the playlist changes during a long drag (another client, or the current track advances via playback), `startPos` is stale by drop time. The trashcan would delete tracks at the wrong positions.

## Scenario Analysis: Moving Albums Across the "Played/Unplayed" Boundary

### The `groupedPlaylist` computed flags:
- `isPlayed`: `index < currentPos` (per-track)
- `isCurrentTrack`: `index === currentPos`
- `isFullyPlayed`: all tracks have `isPlayed=true` and none are `isCurrentTrack`

### Scenario A: Move fully-played album to after current track
**Example**: [A(played), B(current), C(unplayed)] → drag A after C

- `targetPos` = sum of tracks before A's new position in tempGroups
- MPD correctly moves the range; current track position shifts backward (fewer items before it)
- `fetchPlaylist()` gets the updated `currentPos` from MPD
- Album A now appears as "unplayed" (all its tracks are after currentPos)
- **Result: Correct** ✓

### Scenario B: Move unplayed album to before current track
**Example**: [A(played), B(current), C(unplayed)] → drag C before A

- MPD moves C's range to position 0; current track position shifts forward (items inserted before it)
- `fetchPlaylist()` gets the updated `currentPos`
- Album C now appears as "played" (all its tracks are before currentPos)
- **Result: Correct** ✓

### Scenario C: Move the album containing the current track
**Example**: [A(played), B(current at track 2 of 3), C(unplayed)] → drag B after C

- MPD moves B's range (including the currently playing track); MPD internally updates the current song position to the new location
- `fetchPlaylist()` gets the updated `currentPos`
- The moved album retains its `hasCurrentTrack` flag and `isPlayed` reflects the new position
- **Result: Correct** ✓

### Scenario D: Move album across current track boundary (adjacent)
**Example**: [A(played, 2 tracks), B(current at pos 2), C(unplayed, 3 tracks)] → drag A after B

- `targetPos` = 2 (B's track count) = position right after current
- MPD `move 0:2 2`: removes A, shifts B to 0, then inserts A at position 2 → [B, A, C]
- Current track was at pos 2, shifted to pos 0 after removal, stays at 0 after insertion at pos 2
- Album A now straddles the current position boundary: some tracks before, some after
- **Result: Correct** ✓ (MPD handles position tracking)

### Scenario E: "Clear Played" during playback
- Collects positions from `isFullyPlayed` albums
- Removes in descending order via `removeMultipleFromPlaylist` (sequential, awaited)
- MPD adjusts current position for each removal
- Final `fetchPlaylist()` gets the correct state
- **Result: Correct** ✓

---

## Summary of Fixes Needed

| Priority | Bug | Fix |
|----------|-----|-----|
| **Critical** | `removeAlbum` parallel un-awaited deletes | Replace with `removeMultipleFromPlaylist(group.tracks.map((_, i) => group.startPos + i))` |
| **Moderate** | Cross-drag race with mutated tracks arrays | Compute `targetPos` from `mpdStore.playlist` directly instead of relying on cached group objects, or add a guard to prevent album drag while a track move is pending |
| **Low** | Visual flicker from empty setter | Implement optimistic update in the setter, or debounce the computed re-evaluation during drag |
| **UX** | Cross-album track drag breaks grouping | Consider preventing cross-album track drags (`group="tracks"` scope isolation per album), or accept and document the behavior |
| **Edge** | Stale `draggedItem` on trashcan drop | Re-validate positions against current playlist state before executing the deletion |
